# Multi-format Motion Workbench Architecture

Date: 2026-06-13
Status: P1 infrastructure proposal with an isolated SVGA inspection adapter.

## 1. Goal and boundary

Auto SVGA may evolve from an avatar-frame SVGA pipeline into a local motion
asset workbench. The near-term architectural goal is to make future format work
additive instead of placing more format-specific branches inside the CLI, Web
preview page, or current avatar-frame project protocol.

This proposal covers:

- SVGA
- VAP
- Lottie
- animated WebP
- WebM
- APNG
- sprite sequences
- playback, parsing, replaceability, conversion, export, specification checks,
  and performance checks

This proposal does not:

- add a new top-level product mode
- add a player, parser, encoder, or dependency
- convert an existing asset
- change `project.json`, SVGA export, or Web preview behavior
- expand the current production scope beyond `avatar_frame`
- claim cross-format semantic conversion

## 2. Current architecture audit

The current code has two overlapping project models:

- `src/mvp/types.ts` drives the active avatar-frame planning, preview, report,
  SVGA export, and packaging chain.
- `src/types/project.ts` and `src/exporters/` support the older example route.

Important coupling:

1. `src/exporters/exporter.ts` accepts `AvatarFrameProject` directly. It is an
   SVGA-era exporter abstraction, not a reusable format adapter.
2. `src/mvp/svga-exporter.ts` owns protobuf encoding, image registration,
   per-frame expansion, and SVGA validation in one format-specific module.
3. `src/mvp/preview-renderer.ts` owns raster rendering and directly invokes the
   system `ffmpeg` command for WebM and MP4.
4. `tools/svga-player-preview/main.js` combines product UI state, artifact
   discovery, format inspection, and playback control.
5. The Web page loads `pako` and `svgaplayerweb` from a CDN. This is acceptable
   for the current local validation page but not for an offline desktop client.
6. `tools/svga-player-preview/server.mjs` uses Node filesystem and HTTP APIs,
   so it is a host adapter rather than portable workbench core.

The current SVGA pipeline should remain intact. New workbench contracts live in
`src/workbench/` and must not be imported by production commands until one
format migration has its own acceptance task.

## 3. Capability matrix

Maturity meanings:

- **Current**: implemented in the repository today.
- **Partial**: a narrow route exists, but it is not a general workbench capability.
- **Planned**: architecture is clear enough to schedule.
- **Research**: upstream maintenance, format semantics, or licensing must be resolved.
- **Unsupported**: intentionally outside the proposed capability.

### Format-specific boundaries

#### SVGA

Keep the current standard protobuf and zlib route. `FrameEntity` does not have a
frame-level `imageKey`; image replacement remains sprite-level. The first
workbench migration should wrap existing inspection and playback behavior
without changing generated bytes.

#### VAP

VAP is a video/container delivery format with platform playback concerns, not
an interchangeable layer model. Treat its fusion/replacement data as
format-specific metadata. Do not promise VAP export until an encoder,
redistribution model, and desktop playback path are approved.

Research note: `docs/research/vap-official-repository-research.md` records the
2026-07-04 official Tencent VAP repository findings, including the stopped
upstream maintenance state, `vapc` config model, fusion-mask semantics, 1504
compatibility limit, and follow-up work package candidates.

#### Lottie

Lottie contains vector, text, shape, mask, expression, and external asset
semantics that a raster frame sequence cannot preserve. Rasterizing Lottie is a
preview/conversion fallback, not a lossless format conversion.

#### WebP, WebM, and APNG

These are flattened outputs. They can participate in playback, measurement,
specification checking, and frame-sequence conversion. They cannot expose
semantic layers or replaceable sprites after flattening.

#### Sprite sequence

Use a lossless raster interchange consisting of ordered RGBA frames plus an
explicit timing manifest. Never infer production timing solely from filenames.
This intermediate is useful for flattened formats but must not replace the
existing avatar-frame `project.json` source of truth.

## 4. Proposed layers

```text
Product workflows
  inspect | compare | validate | recommend | convert | export
                         |
Workbench application services
  adapter registry | playback coordinator | spec service | recommendation service
                         |
Format ports
  FormatAdapter | PlaybackAdapter | MotionSpecChecker | ExportPipeline
                         |
Format implementations
  SVGA | VAP | Lottie | WebP | WebM | APNG | SpriteSequence
                         |
Host ports
  file access | cache | temporary workspace | process runner | telemetry
                         |
Hosts
  CLI Node host | Web validation host | future desktop host
```

Rules:

1. Core contracts do not import Node filesystem, DOM, Canvas, Electron, or Tauri.
2. `FormatAdapter` inspects and parses. It does not render UI or write files.
3. `PlaybackAdapter` owns a disposable playback session for one host target.
4. `ExportPipeline` produces bytes and metadata. The host decides where to save.
5. Long operations receive cancellation and progress callbacks.
6. Expected failures return structured issues. Programmer errors may still throw.
7. Format-specific metadata stays under `metadata`; common UI reads only
   `MotionAssetInfo`.
8. Recommendation results explain tradeoffs and never silently convert.

The initial contracts are in:

- `src/workbench/contracts.ts`
- `src/workbench/capabilities.ts`

They are deliberately not connected to the existing runtime.

The first implementation slice is in `src/workbench/svga/`:

- `format-adapter.ts` maps inspected SVGA bytes to `MotionAssetInfo`
- `types.ts` defines the format-specific inspection boundary
- `node-protobuf-inspector.ts` contains Node zlib and protobuf loading

The adapter has no Node, DOM, Canvas, filesystem, or browser dependency. The
Node inspector is injected and may later be replaced by a browser or desktop
host implementation. No production command imports this adapter yet.

## 5. Core interfaces

### `FormatAdapter`

Responsibilities:

- cheaply probe a source
- parse common metadata
- enumerate layers and resources where the format supports them
- retain format-specific facts in metadata

It must not:

- create a player
- decide a target format
- mutate source files
- assume a local filesystem path

### `MotionAssetInfo`

The read model shared by overview, asset inspection, specification checks, and
recommendations. Optional fields represent genuine format differences. A
flattened WebM must not fabricate layers to look like SVGA.

### `PlaybackAdapter`

Creates a `PlaybackSession` bound to a host target. A session owns load, play,
pause, seek, loop, replay, state, and disposal. Synchronization between two
sessions belongs to a playback coordinator, not to either format adapter.

### `MotionSpecChecker`

Checks a normalized asset against a named delivery specification. Checks should
be deterministic and side-effect free after parsing. Runtime playback quality
is reported separately and must not be inferred from parse success.

### `FormatRecommendationEngine`

Ranks formats from explicit platform and priority inputs. It must include
reasons and warnings. A recommendation is advice, not an export action.

### `ExportPipeline`

One target format per implementation. It accepts either a semantic
`MotionAssetInfo` when supported or a `FrameSequenceIntermediate` for flattened
conversion. It does not write directly to arbitrary host paths.

### `FrameSequenceIntermediate`

RGBA frames, dimensions, per-frame duration, loop, alpha mode, and color space.
This is a conversion bridge for raster outputs only. It is not a universal
replacement for Lottie vectors, SVGA sprites, or VAP format metadata.

## 6. Typical flows

### Inspect and play

```text
source -> registry.probe -> FormatAdapter.parse -> MotionAssetInfo
                                      |
                                      -> PlaybackAdapter session -> host target
```

### Specification check

```text
MotionAssetInfo + MotionSpec -> MotionSpecChecker -> report
```

### Flattened conversion

```text
source adapter -> renderer/decoder -> FrameSequenceIntermediate
                                  -> target ExportPipeline -> artifact
```

The conversion step must state whether it is:

- semantic and lossless
- rasterized but visually targeted
- lossy or feature-reduced

### Recommendation

```text
MotionAssetInfo + target platforms + priorities + required capabilities
  -> ranked formats with reasons and warnings
```

## 7. Desktop-client preparation

The workbench core should remain host-neutral. A future desktop host supplies:

- local file handles and save dialogs
- temporary workspace and cache lifecycle
- process execution policy
- bundled or native playback surfaces
- network policy and update policy
- crash-safe cancellation and cleanup

Do not choose Electron or Tauri in this architecture task. The deciding spike
must compare:

- player compatibility and Canvas/video behavior
- binary size and memory
- Node/process access requirements
- code-signing and sandbox implications
- offline dependency bundling
- native codec availability

Current client blockers:

1. CDN-only SVGA and pako scripts.
2. archived SVGA Web upstream.
3. direct system `ffmpeg` invocation.
4. browser globals and UI logic mixed with playback state.
5. Node paths used as command inputs rather than host-neutral sources.

### 7.1 Short-term compatibility guardrails

The current owner-visible product remains the short-term macOS SVGA client.
VAP, Lottie, Windows, and AEB package intake are not short-term visible scope.
However, the short-term UI/UX rebuild and client implementation should remove
avoidable coupling that would otherwise multiply later migration cost.

Allowed now, as non-visible infrastructure discipline:

1. Route open, drag, save, overwrite, recent-file, menu, compare, settings,
   logs, and appearance actions through host command boundaries.
2. Treat the preview stage as a playback surface driven by a playback model.
   The SVGA player is the current implementation, not the UI contract.
3. Render overview facts, asset rows, replaceable rows, thumbnails, metric
   actions, and empty/error states from normalized inspection and capability
   models wherever practical.
4. Keep unsupported-format, parse-failure, playback-failure, dependency-missing,
   file-unreadable, and capability-unsupported feedback on a shared typed path.
5. Keep local path display, redaction, recent records, save targets, temporary
   files, and error details behind host/path helpers that can later support
   Windows path and permission differences.
6. Keep command identifiers platform-neutral so macOS menu entries can become
   Windows menu, toolbar, shortcut, or command-palette entries without changing
   product logic.
7. Keep all player/parser dependencies offline-bundled and subject to license,
   maintenance, bundle-size, codec, and redistribution review before use.

Not allowed now:

- visible VAP, Lottie, Windows, AEB, import-package, or format-selection
  controls in the short-term app;
- partial hidden playback paths that accept unsupported formats without product
  acceptance;
- inactive future-format placeholders;
- new dependencies for Lottie, VAP, FFmpeg, video codecs, AI, or external
  services without a separate approved task;
- claiming Windows readiness or multi-format support from structural cleanup.

## 8. Dependency and license review

No dependency is added by this proposal. Candidates require a separate
approval task with a lockfile diff, bundle-size measurement, security review,
and redistribution plan.

| Candidate | Use | License | Current concern |
|---|---|---|---|
| `svgaplayerweb` | SVGA Web playback | Apache-2.0 | Upstream archived in 2023; existing CDN load is not offline-safe |
| `lottie-web` | Lottie Web playback | MIT | Renderer feature differences and bundle size need a spike |
| Tencent VAP | VAP playback/reference | MIT | Upstream states it is no longer maintained |
| `UPNG.js` | APNG parse/encode candidate | MIT | Latest GitHub release shown as 2020; maintenance and pako dependency need review |
| FFmpeg | WebM/video conversion | LGPL-2.1-or-later or GPL-2.0-or-later by build | Bundled builds, codecs, source offer, and x264/GPL choices require legal approval |
| Browser Canvas/ImageBitmap/video | raster and video playback | platform API | Capability and codec support vary by host |

Primary references checked on 2026-06-13:

- <https://github.com/svga/SVGAPlayer-Web>
- <https://github.com/airbnb/lottie-web>
- <https://github.com/Tencent/vap>
- <https://github.com/photopea/UPNG.js>
- <https://ffmpeg.org/legal.html>

Additional VAP-specific research checked on 2026-07-04:

- `docs/research/vap-official-repository-research.md`

FFmpeg is the highest distribution risk. The current MP4 preview command uses
`libx264`, which the FFmpeg legal checklist identifies as GPL-related. Do not
bundle that toolchain into a proprietary desktop client without an approved
build and compliance plan.

## 9. Staged rollout

### Phase 0: contracts and evidence

- keep contracts isolated
- maintain the capability matrix
- add dependency decision records before installation
- define fixture policy using generated mock assets

Exit: contracts compile, capability claims match current code, no behavior change.

### Phase 1: wrap SVGA inspection

- [x] implement an SVGA `FormatAdapter` by reusing current protobuf decoding
- [x] return `MotionAssetInfo`
- [x] add adapter metadata parity tests
- [ ] choose the first production inspection call site in a separate task
- do not change exporter bytes or player UI

Current evidence: synthetic standard SVGA bytes preserve dimensions, FPS,
duration, image keys, Sprite references, and counts; counts also match the
existing MVP SVGA validator. Production integration remains intentionally
deferred.

### Phase 2: extract playback sessions

- wrap current SVGA player behind a browser `PlaybackAdapter`
- add native WebM reference adapter
- move synchronized playback into a coordinator
- bundle approved dependencies locally

Exit: current local/export review behavior passes visual regression and offline loading.

### Phase 3: specifications and performance

- introduce versioned delivery specifications
- add deterministic static checks
- measure runtime metrics separately
- keep technical and visual acceptance distinct

Exit: checks are reproducible and do not claim visual success.

### Phase 4: sprite sequence interchange

- define manifest schema
- implement frame cache limits and cancellation
- add APNG/WebP experiments behind explicit flags

Exit: lossless frame round-trip fixtures and memory-budget tests pass.

### Phase 5: format spikes

- Lottie playback and feature inventory
- VAP playback and container inventory
- desktop host comparison
- recommendation rules based on measured data

Exit: ADR for each approved dependency and host direction.

### Phase 6: approved conversion/export

Only add a target pipeline after:

- license and redistribution approval
- deterministic fixtures
- size, quality, alpha, and timing comparisons
- cancellation and cleanup tests
- explicit loss disclosure

## 10. Verification strategy

Architecture-level:

- TypeScript strict compilation
- format registry uniqueness
- every format covers all capability categories
- only existing behavior is marked `current`

Future adapter-level:

- probe false-positive fixtures
- malformed input and size-limit tests
- parse metadata parity
- session disposal and cancellation
- loss classification for conversion
- deterministic spec reports
- memory and temporary-file cleanup

Existing regression:

- current MVP tests remain unchanged and pass
- Web preview JavaScript syntax remains valid
- no generated or real design assets are committed

## 11. Explicit non-decisions

- desktop framework
- universal motion document model
- Lottie authoring
- VAP encoder
- bundled FFmpeg
- cross-format replacement semantics
- multi-asset-type product expansion

These require separate evidence and ADRs.
