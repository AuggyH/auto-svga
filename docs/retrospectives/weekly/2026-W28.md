## High-value Documents

### Fast Routing

Use `PRODUCT_DOCUMENTATION_SYSTEM.md` as the routing table and
`PRODUCT_ROADMAP.md` as the main PRD. These two prevent the most expensive
failure mode: rebuilding scope from memory or older prototype assumptions.

### Current UI Truth

Use `DESIGN.md` plus `SHORT_TERM_UI_UX_REDESIGN_EXECUTION_PLAN.md` to keep UI
work tokenized and traceable. They are useful because they tell agents where
visual decisions belong and which old surfaces must not return.

### Mid-term And AEB Boundary

Use `MID_TERM_IMPLEMENTATION_PREP.md` and `AE_BRIDGE_PRODUCT_BRIEF.md` to keep
parallel lanes from colliding. The key value is not more planning; it is the
explicit boundary that mid-term can continue in isolated foundations while AEB
gets priority when capacity conflicts arise.

### Failure Learning

Use P6 retrospectives as an anti-pattern library:

- do not split work only by technical layer;
- do not equate worker PASS with milestone PASS;
- do not let packet/evidence generation become a substitute for workflow
  proof;
- run vertical integration checks after each meaningful slice;
- separate machine proof, owner visual judgment, packaging readiness, and final
  acceptance.

### Token And Review Hygiene

Use `context-budget`, `verification-budget`, `diff-first`, and
`caveman-report` as active execution shortcuts, not optional reading. They
directly address broad scans, long logs, full diffs, and over-verification.

## Repeated Cost Patterns

### 1. Micro-review Overhead

The UI/UX lane produced many narrow polish/split reviews. This improves
traceability, but it also multiplies task-start, smoke, screenshot, review, and
handoff cost.

Rule: batch adjacent visual-only polish into one acceptance bundle when it
shares the same surface, evidence path, and risk level. Keep separate tasks for
behavior, save/output, host security, accessibility-critical focus behavior, or
fragile cross-state layout.

### 2. Evidence Without Acceptance Closure

Many reviews correctly say smoke is regression-only and foreground evidence is
still needed. Repeating that disclaimer is useful once, but costly when it
appears in many adjacent slices without a shared foreground acceptance plan.

Rule: maintain one active foreground-validation checklist per UI/UX bundle.
Individual reviews should link to it instead of restating the same acceptance
boundary every time.

### 3. Technical Split Before Vertical Flow

P6 repeatedly showed that technical-layer work can pass while user journeys
remain unproven. The same risk applies to current short-term and mid-term
integration if implementation, UI, host, smoke, and evidence lanes move without
one vertical owner per user flow.

Rule: each major capability needs a named vertical flow owner and one
end-to-end proof path before the lane declares evidence-ready.

### 4. Historical Context Reload

Many documents are valuable as lineage, but not every task needs them. Broad
repo scans are useful for the first retrospective; they should not become the
default for every task.

Rule: task start should load the smallest active-doc pack:

1. `AGENTS.md`
2. `docs/product/PRODUCT_DOCUMENTATION_SYSTEM.md`
3. `docs/product/PRODUCT_ROADMAP.md`
4. relevant lane doc
5. `docs/retrospectives/PROJECT_EXPERIENCE_GUIDE.md`
6. only the specific review/evidence files named by the task

### 5. Validation Cost Not Always Matched To Risk

The repo already has `verification-budget`, but the review pattern still
suggests frequent repeated smoke/evidence notes. Full or repeated validation is
right for parser/exporter/playback/host changes; visual-only token adjustments
should use targeted checks plus a bundled foreground pass.

Rule: write the validation tier before running checks. If the task is Tier 0
or low-risk UI polish, avoid full regression unless it protects an identified
fragile state.

## Immediate Execution Rules

1. Use a small active-doc pack before work; do not rescan history unless the
   task is a retrospective, migration, or disputed scope decision.
2. Batch adjacent visual-only UI polish by page state or surface.
3. Keep one foreground validation checklist per UI/UX bundle.
4. Do not create a full review package after every small fix.
5. For product behavior, use vertical user-flow work packages, not technical
   layer ownership.
6. Distinguish implementation complete, evidence-ready, owner review,
   accepted, packaged, and released.
7. Record exact token counts only when available; otherwise mark unavailable
   and capture cost drivers.
8. Promote repeated lessons into `PROJECT_EXPERIENCE_GUIDE.md`; keep
   one-off observations in `PROJECT_LESSONS_CANDIDATES.md`.

## Next Recommended Project Moves

1. UI/UX lane: pause new tiny polish slices after the current visual surface
   set, then run a bundled foreground pass across Launch, Preview, Replaceable,
   Optimization, Compare, Settings, and Edit reserved.
2. Short-term implementation lane: focus on S1-S16 functional closure and
   real output flows, especially open/play, inspection, imageKey rename,
   replacement preview, optimization output, save/reopen, and recent files.
3. Mid-term lane: continue isolated M1 compile/edit-session integration only
   where it can hand compiled bytes back to Preview without changing the
   formal short-term surface.
4. AEB lane: keep priority above AI/ComfyUI and use the bridge brief as the
   routing doc; do not let mid-term template authoring absorb AEB capacity.
5. PM lane: maintain the PRD and product-document map, but avoid creating new
   docs unless they replace a recurring decision gap.

## Candidate Lessons To Promote

- Adjacent visual polish needs bundle-level acceptance, not one full overhead
  cycle per tiny UI slice.
- Foreground evidence should be centralized per UI bundle.
- P6 retrospectives should be used as an anti-pattern checklist before
  starting any large multi-lane milestone.
- Validation tier should be declared before running tests.
