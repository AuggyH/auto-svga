# Review: 0.2-WP6 Multi-format Desktop Integration And Internal Alpha

## 1. Summary

Implemented the formal `0.2-multiformat-preview` desktop product mode on the canonical QA-accepted `0.1` UIUX/HB host source, then reapplied the accepted 0.2 multi-format source chain without modifying the frozen WP5 branch/head.

WP6 wires local desktop open/menu/drag flows for SVGA, Lottie JSON, and VAP/MP4 into the accepted WP5 owner-visible candidate model. It reuses the existing short-term shell layout and command state, adds a gated 0.2 preload/API bridge, adds a main-process hidden multi-format session with bounded reads and source-side playback contracts, and adds a renderer controller for overview, facts, replaceable image/text/fusion candidates, playback/recovery controls, and runtime preview replacement/reset.

This is an internal 0.2.0-alpha desktop integration candidate. It is not `0.1.x` visible support, not a package promotion, not real-material visual acceptance, not Product Owner implementation acceptance, and not release readiness.

## 5. Desktop Integration Notes

- The 0.2 mode deliberately reuses the short-term shell HTML/CSS entry so UIUX/HB state, window sizing, theme, resize, menu, and recovery conventions remain aligned with the canonical `0.1` host base.
- The 0.2 mode has its own preload bridge. It does not expose `openSvgaFile`, save, export, overwrite, optimization, or legacy prototype APIs.
- The main menu is product-mode specific: `0.1` keeps `打开 SVGA...`; `0.2` uses `打开预览候选...` and enables only preview/replacement/recovery commands.
- The source-side player targets are still hidden contract targets. Real DOM/WebGL/canvas visual proof is intentionally not claimed.

## 10. Project retrospective

- Value assessment: High.
- Cost drivers: integrating the accepted multi-format chain onto the canonical `0.1` UIUX/HB host base, while preserving formal `0.1` API isolation and avoiding stale host ancestry.
- Avoidable costs: future desktop mode additions should reserve a product-mode bridge seam earlier, so menu/preload/window mode tests do not need as many textual assertion updates.
- Product lesson: a formal 0.2 desktop mode can be owner-visible as a candidate while still withholding package readiness, support, release, and visual-success claims.
- Technical lesson: product-mode-specific preload bridges are safer than expanding one formal bridge with optional future APIs.
- Design / interaction lesson: sharing the short-term shell keeps light/dark, resize, launch, menu, and recovery behavior coherent across 0.1 and 0.2 without forking three format clients.
- Process lesson: source-preserving replay onto the canonical UIUX/HB base kept PM authority, QA-accepted 0.1 behavior, and multi-format chain provenance auditable.
- Follow-up candidate for `docs/retrospectives/PROJECT_LESSONS_CANDIDATES.md`: Yes.
