Implemented a formally gated `Auto SVGA 0.2.0-alpha.1` owner-visible preview candidate state model on top of the CR-approved WP4 hidden workspace. ## 7. Risks and limitations

- The WP5 candidate is source/model evidence, not an installed owner local app and not a promoted package.
- Real-material evidence is limited to read-only inventory categories. No real SVGA/Lottie/VAP material was committed, opened in foreground, captured, or used for a visual success claim.
- SVGA runtime replacement in WP5 uses a controller boundary so future visible-shell integration can bind the existing 0.1 replacement pipeline without merging stale host ancestry.
- VAP visual playback evidence remains absent for non-confidential material; the runtime path remains a hidden/local/reversible spike.
- Lottie replacement is limited to supported top-level image asset ids and text layer identifiers. Unsupported Lottie features, fonts, unsafe paths, and non-inline image replacements fail closed.

## 9. Project retrospective

- Value assessment: High.
- Cost drivers: WP5 crosses Lottie replacement mutation, VAP runtime fusion parameters, SVGA replacement semantics, lifecycle race guards, and product/packaging language boundaries.
- Avoidable costs: Future visible-candidate milestones should define replacement target identifiers before the candidate model is written.
- Product lesson: A `0.2.0-alpha.1` candidate mode can be owner-visible in product identity while still forbidding `0.1.x` exposure, package promotion, and support claims.
- Technical lesson: Runtime replacement should be modeled as a preview-only reload/remount state with generation guards; it must not be mixed with persistent save/export contracts.
- Design / interaction lesson: One right-panel model can present base facts, Lottie text/image candidates, and VAP fusion tags without introducing format-specific clients.
- Follow-up candidate for `docs/retrospectives/PROJECT_LESSONS_CANDIDATES.md`: Yes.
