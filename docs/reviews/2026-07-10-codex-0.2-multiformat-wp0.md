## 7. Fixture Strategy

General rules:

- Commit only synthetic, minimal fixtures or generated fixtures whose license and source are documented.
- Do not commit production VAP, Lottie, SVGA, images, videos, extracted frames, or sequence folders.
- Local-only production samples may be referenced in QA notes by redacted metadata only.
- Fixture generators should create assets at test time where practical.

Lottie fixtures:

- Minimal inline-shape JSON with no external assets: commit under `fixtures/multiformat/lottie/minimal-shape.json` in WP2.
- External local image asset JSON: commit JSON and a tiny generated PNG fixture only if the PNG is programmatic/mock; test deterministic relative path resolution and missing image failure.
- Text layer JSON: commit synthetic text-only JSON to expose text candidates; do not promise persistent text editing.
- Unsupported feature JSON: synthetic expression/mask/effect example that must report `unsupported_feature` without silent playback success.
- Corrupt JSON and wrong media type: small text fixtures for parse failure and unsupported feedback.

VAP fixtures:

- Single-file MP4 with embedded `vapc`: prefer synthetic generated fixture only after tool/license review; until then keep local-only under ignored workspace and record metadata in QA reports.
- Fusion image/text sample: local-only first; later commit only if generated from mock frames and approved.
- Missing `vapc`: tiny placeholder MP4/container fixture or parser-level byte fixture in WP3 to prove fail-closed behavior.
- Corrupt `vapc`: byte fixture with invalid JSON or invalid schema to prove diagnostic path.
- Over-1504 dimensions: config-only or synthetic local fixture that reports compatibility risk without needing production video bytes.
- Adjacent `vapc.json`: diagnostic fallback fixture only; normal support must target embedded `vapc`.

## 8. Adapter And Code Inventory

Reusable:

- `MotionFormat` already includes `svga`, `vap`, and `lottie`.
- `MotionAssetInfo` can represent shared facts: format, name, size, dimensions, timing, layers, resources, metadata.
- `MotionResourceInfo` can carry image/video/vector/audio/font resources and format-specific metadata.
- `FormatAdapter` and `PlaybackAdapter` already separate parse from playback.
- `WorkbenchIssue` already supports typed severity, code, message, path, and details.
- `FORMAT_CAPABILITY_BASELINES` already marks VAP/Lottie as research/planned, not current.

Must stay isolated:

- Current short-term product model is SVGA-specific and sourced from `avatar-frame-inspection-report`.
- Current host open path is `openSvga` and calls `inspectSvga`; it must not be widened in 0.1.
- Current command menu labels are SVGA-only; no VAP/Lottie/import-package/format-selection entry can appear before a 0.2 gate.
- VAP fusion elements must not reuse SVGA `imageKey` semantics.
- Lottie text/assets must not be presented as saveable persistent edits in MVP.
- Playback success must remain separate from parse success.

Low-risk code guard added:

- `src/tests/short-term-command-menu.test.ts` now fails if 0.2 multi-format entry text appears in the 0.1 command menu.

## 9. WP1/WP2/WP3 Implementation Split

### 0.2-WP1: Shared format registry and detection

Priority: first.

Scope:

- Add a 0.2-only registry/probe service.
- Detect SVGA, Lottie JSON, VAP/MP4 candidates, and unsupported inputs.
- Return typed unsupported/missing-dependency/parse-precondition issues.
- Keep 0.1 `openSvga` visible contract unchanged.

Acceptance evidence:

- Unit tests for extension, media type, magic/header, JSON shape, MP4 box probe, unsupported, and redacted paths.
- Regression test that short-term command/menu surface remains SVGA-only.
- No new runtime dependency unless separately approved.

Main risks:

- False-positive JSON or MP4 detection.
- Path leakage in diagnostics.
- Accidentally routing 0.1 open flows through 0.2 detection.

### 0.2-WP2: Lottie JSON parse/playback MVP

Priority: second, because JSON parse and renderer proof are lower risk than VAP.

Scope:

- Parse local Lottie JSON metadata, dimensions, frame rate, duration, assets, text candidates, and unsupported feature markers.
- Prove playback with the approved renderer candidate after dependency approval.
- Support local image assets only with deterministic relative path resolution.

Acceptance evidence:

- Synthetic local JSON fixture plays in a 0.2-gated host.
- Asset/text list renders from normalized `MotionAssetInfo`.
- Unsupported features are visible as warnings/errors.
- Bundle-size/offline report is attached before dependency adoption.
- SVGA 0.1 regression tests remain green.

Main risks:

- Renderer mode differences.
- Fonts, expressions, masks, matte/effects feature gaps.
- External asset path handling.

### 0.2-WP3: VAP `vapc` inspector/playback spike

Priority: third, because VAP has stronger container, playback, and upstream risk.

Scope:

- Inspect local MP4 boxes and extract embedded `vapc`.
- Normalize VAP timing, display/video dimensions, alpha/RGB rectangles, fusion image/text sources, and frame coverage.
- Fail closed on missing/corrupt `vapc`.
- Evaluate local playback path only after dependency/fork/redistribution review.

Acceptance evidence:

- Parser tests for embedded `vapc`, missing `vapc`, corrupt `vapc`, fusion image/text metadata, and over-1504 risk.
- No adjacent `vapc.json` dependency for normal support.
- Playback spike records manual/automated limits without claiming production readiness.

Main risks:

- Unmaintained upstream.
- Browser/Electron video codec and WebGL differences.
- Fusion jitter and platform parity.
- MP4 mutations that strip custom boxes.
