## Implementation

- Added a hidden `0.2` Lottie preview vertical session with explicit gate
  `0.2-hidden-lottie-preview-vertical`.
- Added a host contract that uses local file stat plus bounded range reads for
  detection, inspection, and playback JSON.
- Routed local file-button, menu-open, and drag-drop sources through the WP1
  registry before Lottie-specific inspection.
- Reused WP2A Lottie JSON inspection for normalized version, dimensions,
  timing, layers, image assets, text candidates, and unsupported markers.
- Added a hidden normalized surface with overview, layers, assets, replaceable
  image/text candidates, unsupported markers, playback state, and typed
  failures.
- Added deterministic adjacent image resolution for relative image references.
  Resolved images are inlined as `data:` URIs inside renderer `animationData`
  so the renderer receives no local filesystem path.
- Kept missing, unsafe, oversized, unsupported media, font, embedded, and
  unsupported-feature cases fail-closed before renderer load.
- Added an active request generation guard across local open, registry probe,
  Lottie inspection, adjacent resource resolution, animationData preparation,
  renderer load, recovery, and public playback mutations. Stale async work now
  returns the active model without overwriting active source content, prepared
  playback, or playback session state.
- Added cancellation propagation into `PlaybackSession.load()` and a final
  cancellation/disposed check in the Lottie SVG adapter immediately before
  `renderer.loadAnimation()`. A superseded open that was already inside
  renderer loading can now exit without invoking the renderer or mutating the
  target container.
- Kept disposed hidden sessions terminal for later public controls so page-state
  teardown cannot mutate a disposed model back to `playbackFailed`.
- Extended the approved Lottie SVG adapter with an opt-in
  `allowResolvedImageResources` switch. Default WP2B behavior remains
  animationData-only and still rejects image/font resources.
- Added a hidden Node host adapter for bounded local JSON and adjacent-resource
  reads. It is not imported by the formal `0.1.x` host.

## Repair Contract: MF-LPV-CR-001R

- Lead Implementation Owner: Multi-format implementation lane.
- Evidence Owner: PM/governance for repair-contract evidence check.
- Independent Integration Verifier: Code Review.
- Root-cause hypothesis: the hidden vertical treated `PlaybackSession.load()` as an atomic awaited boundary, but the adapter performs additional asynchronous work inside that call. Once the stale session had been assigned, a newer open could dispose it, yet the stale adapter load still resumed after its renderer dependency promise resolved and called `renderer.loadAnimation()` without checking request cancellation or disposed session state.
- Why the prior fix failed: the generation/session binding added after `await playbackSession.load(...)` prevented stale final model, prepared playback, and active session commits. It did not prevent side effects inside `playbackSession.load()` itself, so stale renderer code could still touch the shared target container before the vertical regained control and discarded the stale result.
- Failure-first test: add an adversarial slow-first/fast-second test where request A reaches and blocks inside renderer loading, request B completes and becomes active, then request A is released. The expected behavior is that the stale A renderer never invokes `loadAnimation()` at all. The existing slow-read overlap and disposed-control regressions must remain.
- Success stop condition: active source, model, and playback session all match the latest request, and the stale source causes zero renderer/container mutation after it is superseded, including when the stale request was already inside renderer loading.
- Failure stop condition: any stale `loadAnimation()` invocation, stale target container/SVG mutation, or a third independent recurrence of `MF-LPV-CR-001` stops implementation and routes a retrospective under `docs/engineering/REPAIR_HEALTH_PROTOCOL.md`.
