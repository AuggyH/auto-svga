## 4.3 Repair Contract: MF-WP7-CR-001R

- Root-cause hypothesis: the generic POSIX path regex is run before the Windows drive-letter regex and is too greedy across mixed text. In a message containing both `/Users/...` and `C:\Users\...`, the POSIX matcher consumes text through the `C` drive letter and stops at the colon, so the later Windows regex sees only `:\Users\...` and cannot match the drive-less fragment.
- Why the prior fix failed: repair 1 moved redaction to the inventory boundary but treated `redactLocalPaths()` as already complete. Its regression checked `/Users/alice`, `Secret Campaign`, and `C:\Users\alice`, but did not check residual `:\Users\alice` or `\Users\alice` fragments after mixed POSIX/Windows replacement.
- Failure-first tests: add a shared helper regression for mixed POSIX + Windows paths and extend the WP7 inventory serialized-output regression to reject `/Users/alice`, `C:\Users\alice`, `\Users\alice`, `:\Users\alice`, `Secret Campaign`, and related sensitive names.
- Success stop condition: direct helper probe and serialized inventory output contain `[local path]` and contain no POSIX path, Windows drive path, drive-stripped Windows user fragment, username, or sensitive project token; owner-preview, formal `0.1` guard, package/lock/media scans, and full regression remain clean.
- Failure stop condition: any residual username-bearing local path fragment, package/lock drift, production media/archive changed file, formal `0.1` visible multi-format leak, or third independent recurrence of `MF-WP7-CR-001` pauses implementation and routes a retrospective under `docs/engineering/REPAIR_HEALTH_PROTOCOL.md`.

## 4.4 Code Review Repair 2

- Finding: `MF-WP7-CR-001R`, mixed POSIX + Windows strings could leave drive-stripped Windows path fragments after shared redaction.
- Repair: `redactLocalPaths()` now redacts Windows drive-letter paths before POSIX paths and then applies a residual Windows-user fragment fallback for `:\Users\...` and `\Users\...` cases. This keeps the shared inventory boundary fix from repair 1 and hardens the helper used by adjacent workbench surfaces.
- Failure-first proof: before the helper repair, the extended WP7 inventory test failed 4/5 with `Missing [local path]:\\Users\\alice\\Desktop\\avatar.png`, and the shared helper test failed 5/6 on the same residual class. Both pass after repair.
- Boundary: no dependency, runtime/player, UI mode, foreground, package, promotion, production asset, or product-support scope changed.

## 10. Project Retrospective

- Value assessment: High
- Cost drivers: cross-format inventory needed to stay purely derived from accepted models while preserving replacement command semantics, right-panel rendering, local-material evidence boundaries, and 0.1 isolation.
- Avoidable costs: VAP fusion asset de-duplication should be treated as a first-order inventory rule whenever fusion tags and generated resources coexist.
- Product lessons: a source-side qualification matrix can make QA planning concrete without turning local materials into committed fixtures or product-support evidence.
- Technical lessons: keep inventory as a derived view model so parser/playback/replacement contracts remain isolated and reviewable.
- Design / interaction lessons: right-panel grouping should show non-applicable format capabilities explicitly instead of leaving users to infer missing sections.
- Process lessons: metadata-only material scans are useful, but their evidence label must stay separate from foreground/visual/package acceptance.
- Repair lesson: derived owner-visible inventory is a redaction boundary even when upstream models are expected to be redacted; sanitize at the final assignment site before setting `pathRedacted=true`.
- Repair 2 lesson: shared privacy helpers need adversarial mixed-path ordering tests, including residual fragments that no longer include the original drive-letter prefix.
