## Root Cause

The short-term renderer only synchronized `launch` / `workbench` sizing when the view changed. The Electron main process did not keep an independent launch-window sizing state during move, resize, or display-metrics events. On a multi-display macOS setup, a launch view could therefore remain visible inside a previously applied workbench-sized window after display movement or display-scale lifecycle events.

QA's scripted `setBounds` movement did not reproduce the issue after forcing `720 x 720`, but the code path had no launch-mode guard after cross-display movement. The fix closes that gap deterministically instead of hardcoding any display role.

## Fix

- Added `short-term-window-bounds-policy.cjs`, a pure bounds policy that preserves the current launch window size inside the runtime-detected target display work area.
- Main process now tracks `shortTermWindowMode` and `shortTermLaunchWindowSize`.
- Launch mode now listens to `move`, `moved`, `resize`, `resized`, and `display-metrics-changed`, then re-applies the preserved launch dimensions if macOS or the renderer leaves the launch page in a wide workbench-sized frame.
- Explicit user resize is still respected through `will-resize`; the preserved launch size updates only for user resize in launch mode.
- Workbench sizing remains restricted to non-launch views after a file is loaded.
- Renderer window-mode sync treats `failed` with no source file as launch sizing, preventing parse/open failure recovery from promoting the empty launch surface into workbench sizing.
- macOS package proof source audit now includes the new bounds policy file.
