## Proposed CSS Naming

R2 does not change code. These are proposed WP1 token names for aligning code
with Figma variables while keeping existing compatibility aliases.

- `基础/色彩/中性色/50` -> `--asv-base-neutral-50`
- `基础/色彩/蓝色/500` -> `--asv-base-blue-500`
- `基础/间距/间距/12` -> `--asv-base-space-12`
- `基础/圆角/圆角/8` -> `--asv-base-radius-8`

Semantic token naming:

- `语义/色彩/文字/主要` -> `--asv-color-text-primary`
- `语义/色彩/表面/画布` -> `--asv-color-surface-canvas`
- `语义/间距/面板/内边距` -> `--asv-space-panel-padding`

`--asv-action` should remain until call sites are migrated.

## Gap List

1. Base color palette gap:
   - Figma defines 32 base color variables.
   - Code has no equivalent `--asv-base-*` color palette.
   - WP1 should add base palette tokens first, then let semantic tokens alias
     them where possible.

2. Semantic color value drift:
   - Several existing semantic tokens have matching intent but different
     values, especially action blue, canvas surface, window surface, borders,
     and status colors.
   - This drift is expected from the pre-Figma visual pass and should be
     resolved deliberately in WP1 instead of by scattered CSS edits.

3. Surface model mismatch:
   - Figma semantic surfaces are mostly solid base-color aliases.
   - Code currently uses translucent panels, `color-mix`, and component-level
     surfaces for macOS-like depth.
   - WP1 should decide which mixed surfaces remain component tokens and which
     must become semantic Figma-aligned tokens.

4. Spacing scale gap:
   - Code lacks exact base spacing tokens for 0, 2, 6, 10, 32, 40, and 48.
   - Several raw component dimensions already imply those values.

5. Radius mode gap:
   - Figma has medium and large radius modes.
   - Code has no radius mode mechanism and currently uses 6/8/8px primitive
     radii, including 6px which is not in the Figma radius scale.

6. Exact semantic tokens missing:
   - `--asv-color-text-disabled`
   - `--asv-color-text-danger`
   - `--asv-color-text-link`
   - `--asv-color-surface-mask`
   - `--asv-color-surface-muted`
   - `--asv-color-border-focus`
   - `--asv-color-action-secondary`
   - `--asv-color-action-hover`
   - `--asv-color-status-success-bg`
   - `--asv-color-status-warning-bg`
   - `--asv-color-status-danger-bg`
   - `--asv-color-drag-accept`
   - `--asv-color-drag-reject`
   - `--asv-radius-control`
   - `--asv-radius-card`
   - `--asv-radius-modal`
   - `--asv-radius-toast`

7. Figma variable scope cleanup candidate:
   - All read variables currently report `ALL_SCOPES`.
   - This does not block code implementation, but it is a future Figma library
     cleanup candidate if the design system matures.

## Code Debt List For WP1

- The code token file is structurally good but not yet Figma-aligned at the
  primitive/base layer.
- Compatibility aliases are useful and should remain, but they can hide whether
  a token is aligned with Figma or only legacy-compatible.
- Dark-mode values are duplicated under media query and explicit
  `[data-appearance="dark"]`; WP1 should avoid increasing this duplication.
- Component tokens sometimes encode values that belong in the base spacing or
  semantic spacing layer.
- Shadow, blur, checker, and mixed-surface effects are not covered by the R2
  Figma variable collections and should stay component/effect tokens unless R4
  component reads prove otherwise.

## WP1 Recommendation

Before visual polish starts, WP1 should:

1. Add Figma base palette tokens under an English CSS namespace such as
   `--asv-base-neutral-50` and `--asv-base-blue-500`.
2. Rebase existing semantic color tokens onto Figma semantic aliases where
   intent is exact.
3. Add missing semantic tokens rather than overloading existing status/action
   tokens.
4. Add full base spacing and radius scales.
5. Keep existing compatibility aliases and component tokens stable while the
   page-state UI is migrated.
6. Preserve the design-system check and update its raw-value allowances only
   when WP1 actually changes CSS.

R2 is complete once this document, the call log, and the task retrospective are
committed. No CSS should be changed as part of R2.
