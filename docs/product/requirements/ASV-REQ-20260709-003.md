## Feasibility And Decision

- Research or feasibility summary: Existing architecture already defines `MotionFormat`, `MotionAssetInfo`, capability baselines, and a multi-format adapter direction. VAP official research identifies `vapc` extraction, fusion element display, 1504 limit, and playback/tooling risk. Lottie support is likely lower risk for preview but still needs renderer and feature compatibility review.
- Dependencies:
  - Lottie renderer candidate review, likely `lottie-web`, before adoption.
  - Tencent VAP Web/reference implementation review before adoption.
  - Safe local fixtures for Lottie JSON and VAP MP4 with `vapc`.
- Risks:
  - VAP upstream maintenance risk and MP4/codec/WebGL behavior differences.
  - Lottie renderer feature gaps, external asset resolution, font differences, and bundle-size impact.
  - Partial support could confuse `0.1.x` users if exposed too early.
- Alternatives considered:
  - Add VAP/Lottie directly to the current `0.1.x` app: rejected because it would break the SVGA Preview MVP freeze and hide future-format risk inside the owner baseline.
  - Keep VAP/Lottie as long-term only: rejected because Product Owner wants this work to start sooner.
- Final decision: Start `0.2.0-alpha.1` with a readiness/adapter spike and keep all visible multi-format behavior gated out of `0.1.x`.
