## Product Scope

- PRD anchor: `docs/product/PRODUCT_ROADMAP.md`, `Long-term Direction`
- Related product brief or review: `docs/product/AE_BRIDGE_PRODUCT_BRIEF.md`, `docs/product/MACOS_SVGA_WORKBENCH_FOUNDATION_CONTRACT.json`
- Target product version: Auto SVGA `0.6.x+`
- Product phase: Other
- Requirement type: Production workflow
- Current behavior or limitation: Auto SVGA has deterministic build/export foundations, design rules, templates, asset skills, and early bridge work but no Agent-facing Motion Plan API or ComfyUI orchestration.
- Intended behavior: Let an Agent translate a request with Design Rules and asset skills, orchestrate a trusted local/LAN/self-hosted ComfyUI workflow, produce a validated Asset Pack and Motion Plan, execute a transactional headless engine, export, and hand the result to the desktop client for human acceptance.
- User value: Automate repeatable motion-asset creation without forcing the Agent to drive a complex human editor.
- Explicit non-goals: public-cloud provider dependency as the baseline, autonomous production acceptance, arbitrary asset categories in the first vertical, hidden source mutation, or UI-first architecture.

## Feasibility And Decision

- Research or feasibility summary: Existing Design Rules, template/asset-skill direction, project schema, exporters, preview, validation, AEB package facts, and shared format work provide partial foundations. The first ComfyUI MVP is being developed separately and will be synchronized later.
- Dependencies: external ComfyUI MVP handoff; versioned Asset Pack and Motion Plan schemas; headless engine API; trusted endpoint/auth/data-lifecycle contract; deterministic workflow receipts; avatar-frame material and quality corpus.
- Risks: nondeterministic generation, asset provenance/licensing, privacy, remote endpoint trust, large intermediates, quality assessment, model/workflow drift, retries, and false automated acceptance.
- Alternatives considered: Agent drives the desktop UI, rejected because it couples automation to human interaction and fragile foreground state; start with many asset categories, rejected in favor of one avatar-frame vertical.
- Final decision: Architect Agent-first. Desktop UI and Agent API call the same headless engine; keep implementation deferred until prerequisites are real.
