## Product Scope

- PRD anchor: `docs/product/PRODUCT_ROADMAP.md` S3, S8, S17, S18 and runtime structure estimation / optimization sections.
- Related product brief or review:
  - `docs/retrospectives/production-cases/2026-07-07-lucky-notice-svga-runtime-structure-memory.md`
  - `docs/reviews/2026-07-07-codex-lucky-notice-runtime-structure-case.md`
- Product phase: Short-term
- Requirement type: Feature / Optimization / production performance diagnostic
- Current behavior or limitation: A real production SVGA can have small encoded size and decoded image memory while expanding to thousands of sprites and hundreds of thousands of frame records in mobile runtime. If the app only shows file size, decoded image memory, and image count, it can falsely present the file as low risk.
- Intended behavior: The short-term client must detect and display runtime structure risk separately from decoded image memory, and must provide output-producing safe optimization for supported runtime-structure problems such as all-zero sprite/frame pruning and newly unreferenced image cleanup. Risky or target-player-specific structure optimizations remain review-only until validated.
- User value: Prevents mobile performance issues from being pushed back to designers or client engineers when Auto SVGA can detect and address the structural cause.
- Explicit non-goals:
  - Do not claim exact measured mobile memory without target-player measurement.
  - Do not silently apply target-player-specific low-alpha thresholds to generic SVGA compatibility.
  - Do not implement full visual rebaking or sequence-fanout collapse unless playback comparison and output validation are proven.
  - Do not expose unsupported optimization controls in the formal short-term client.
  - Do not commit production SVGA files or generated optimized outputs.

## Feasibility And Decision

- Research or feasibility summary: The lucky-notice case established measurable signals: sprite count, FrameEntity count, alpha-positive/invisible ratios, sequence-frame fanout, and per-frame visible density. Safe all-zero pruning and newly unreferenced cleanup are bounded. Low-alpha pruning, FPS resampling, sequence-fanout pruning, and rebake/collapse need review gates or target-player profiles.
- Dependencies:
  - Existing SVGA inspection/parser capability.
  - Short-term right-information rendering for S3/S8/S17.
  - Short-term optimization workflow for S18 output bytes and comparison.
  - QA real-material and synthetic high-fanout test coverage.
- Risks:
  - Runtime structure memory is an estimate until target-player allocation profiles are calibrated.
  - Some structure optimizations are visually or player-behavior sensitive.
  - Large production files may require careful local-only evidence handling.
- Alternatives considered: Leave this as design-side cleanup guidance or client-engineering debugging. Rejected because the issue is deterministic and measurable in the SVGA file structure.
- Final decision: Implement S17 diagnostics and S18 safe structure optimization in the short-term client, with risky target-player-specific items review-only.
