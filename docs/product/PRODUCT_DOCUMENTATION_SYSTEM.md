# Product Documentation System

Date: 2026-07-01
Owner role: Product Manager
Status: active documentation governance for SVGA Workbench v1

## Purpose

This document defines how product information is organized, maintained, and
used to guide Auto SVGA development.

Auto SVGA is already in active development, so the goal is not to create a
large new planning layer. The goal is to keep product direction, scope,
requirements, evidence, release readiness, and follow-up work aligned enough
that engineering can continue without guessing.

## PM Responsibility Model

A product manager owns the product "why" and "what": product direction,
problem framing, user needs, prioritization, requirement clarity, release
readiness, and success evidence. Engineering owns the implementation "how" and
validation details, but product management must make the acceptance boundary
clear before work is called complete.

For Auto SVGA, PM ownership means:

- Keep the product positioning and anti-drift rules current.
- Keep the active mainline and roadmap readable.
- Convert product asks into bounded requirements with success criteria.
- Keep backlog items prioritized by product value, support boundary, evidence
  need, dependency risk, and release timing.
- Keep supported and unsupported capabilities explicit.
- Keep first-distribution readiness separate from product acceptance, signing,
  notarization, Windows distribution, and public release.
- Keep evidence and release artifacts truthful: no `PASS`, accepted, released,
  or visually successful claim without the required proof.
- Turn owner feedback, reviewer findings, real-asset results, and release
  blockers into roadmap, backlog, decision, or status updates.

PM does not own or bypass:

- source-level implementation choices
- test implementation details
- security exceptions
- generated review packet mechanics
- raw production asset handling
- unverified playback, export, optimization, or visual success claims

## Current Source Hierarchy

When documents conflict, use this order:

1. Latest explicit Product Owner correction or active autonomous status doc.
2. `docs/product/PRODUCT_ROADMAP.md` as the only main PRD authority for current
   product requirements, phase direction, interaction expectations, and
   short/mid/long-term planning.
3. `docs/autonomous/SVGA_WORKBENCH_V1_STATUS.md` for active Workbench v1 run
   state on the autonomous branch.
4. Capability boundary docs and ADRs for specific scope rules.
5. Current-head review/evidence artifacts generated by approved commands.
6. Historical docs, frozen baselines, old review packets, and older status
   files as lineage only.

`docs/CURRENT_STATUS.md`, `docs/CHANGELOG.md`, and `README.md` still contain
useful lineage, but they are not the active Workbench v1 product authority until
they are refreshed.

## Main PRD Authority

`docs/product/PRODUCT_ROADMAP.md` is the single project-level PRD authority.

No other document may redefine the project goal, short-term scope, mid-term
scope, long-term scope, required product capabilities, interaction model, or
release eligibility. Stage-specific or version-specific PRD material must link
back to `docs/product/PRODUCT_ROADMAP.md` and may only add implementation
details, evidence requirements, or acceptance checks for that stage.

If a stage document conflicts with the main PRD, update the stage document or
mark it historical. Do not create a second main PRD.

## Duplication Rule

Before creating a new product or PRD-style document, search existing product,
roadmap, milestone, ADR, backlog, and implementation-plan docs first. Prefer
updating the closest existing source of truth.

Create a new product document only when all are true:

- no existing source has the same ownership or update cadence
- the new document will become a durable source of truth, not a one-off summary
- it is linked from this document and from the relevant roadmap or status doc
- older overlapping material is marked as historical, superseded, or supporting
  context

## Required Product Brief

Use a short product brief before implementing a meaningful capability, release
track, or scope change. It can be a standalone doc or a section inside the
implementation plan.

Required fields:

- Problem: what user or delivery uncertainty this reduces.
- Target user: designer, reviewer, integrator, internal tester, or release
  owner.
- Mainline priority: P0-P7 from `auto-svga-core-guard`.
- Current behavior and limitation.
- Proposed behavior.
- Explicit non-goals.
- Supported and unsupported cases.
- Acceptance criteria.
- Evidence required: tests, desktop smoke, packaged proof, real-asset matrix,
  manual visual review, or release artifact.
- Privacy and asset boundary.
- Rollback path.
- Documents to update when complete.

Small bug fixes may use the task review file instead of a separate brief, but
the acceptance criteria and non-goals still need to be visible somewhere.

## Development Readiness Rule

Before a meaningful product, UI, feature, release, or acceptance task starts,
the relevant main-PRD section must provide enough information for engineering
to avoid guessing.

Minimum required fields:

- requirement ID or named scope section
- target user or user-facing workflow
- current behavior or limitation
- intended behavior
- explicit non-goals
- acceptance criteria or required evidence
- dirty/save behavior when the task can create output
- error, empty, loading, or recovery states when user interaction is involved
- product docs that must be updated when complete

If any of these fields are missing and the task could change product behavior,
the agent should ask the Product Owner or add the missing product clarification
to the authoritative document before implementation.

## Status Vocabulary

Use these states consistently:

- Idea: captured but not evaluated.
- Candidate: useful, but not committed.
- Planned: accepted into a phase or release track.
- Active: implementation or validation is underway.
- Evidence-ready: implementation has current-head evidence, but Product Owner
  acceptance is not claimed.
- Owner review: ready for explicit Product Owner review.
- Accepted: Product Owner or designated acceptance authority approved it.
- Released: shipped through a named distribution tier.
- Deferred: intentionally postponed.
- Blocked: cannot proceed without an external input or product decision.
- Historical: kept only as lineage, rollback reference, or prior evidence.
- Superseded: replaced by a newer current-head source or artifact.

Do not use "basically complete," "PASS," "ready," or "released" unless the
document also names the exact gate that makes the word true.

## Release And Distribution States

Keep these separate:

- Product function implemented.
- Product function validated.
- Review package generated on final head.
- Product Owner accepted.
- D0 internal unsigned package available.
- D1 trusted signed/notarized macOS package available.
- D2 trusted Windows package available.
- Public release.

An internal unsigned App ZIP can be useful evidence, but it is not a production
release and does not imply Product Owner acceptance.

## Product Metrics And Evidence

Auto SVGA is local-first and should not add telemetry just to satisfy product
measurement. Use deterministic and reviewer-visible evidence:

- packaged app open/load/playback success
- Save As and reopen success
- inspection report availability
- Motion Asset Audit availability
- optimization candidate counts and saved-copy proof
- replacement support, reset, save behavior, and reopen proof
- undo/redo proof only when a phase explicitly promotes undo/redo
- sequence repair supported-case counts and fail-closed counts
- redacted real-asset validation matrix coverage
- validation command pass/fail summary
- package manifest, hash, size, privacy audit, and startup proof
- owner feedback closure map
- known limitations and unsupported cases

If later user analytics are desired, they need a separate privacy and product
decision. They are not part of the current first distributable version.

## Maintenance Cadence

- Task start: read the product roadmap, active status, relevant boundary docs,
  the requested task review/evidence path, and relevant sections of
  `docs/retrospectives/PROJECT_EXPERIENCE_GUIDE.md`.
- During implementation: update only the docs directly affected by the change.
- Task completion: update the task review retrospective section and append a
  compact entry to `docs/retrospectives/TASK_RETRO_LEDGER.jsonl` for
  meaningful tasks.
- Weekly: aggregate recent reviews, task-ledger entries, commits, blockers,
  and token-cost patterns into `docs/retrospectives/weekly/YYYY-WW.md`.
- Monthly or after four weekly reviews: distill repeated lessons into
  `docs/retrospectives/monthly/YYYY-MM.md` and
  `docs/retrospectives/PROJECT_EXPERIENCE_GUIDE.md`.
- Meaningful checkpoint: update active status, roadmap/backlog if scope moved,
  and regenerate evidence only when the checkpoint is worth a review package.
- Before release candidate: update distribution prep, known limitations,
  release notes, rollback instructions, and package evidence.
- After owner feedback: update the active status, backlog, finding ledger or
  decision record, and any affected release gate.
- After acceptance/release: update changelog/status docs and mark superseded
  evidence as historical.
